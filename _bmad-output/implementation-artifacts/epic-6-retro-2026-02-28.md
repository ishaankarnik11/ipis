### AI-1: Sprint-Status Bypass When Implementation Runs Outside Dev-Story Workflow

**Discovered:** 2026-02-28, during Story 6.1 implementation
**Severity:** MEDIUM (process gap, not data loss)
**Affected Story:** 6-1-project-dashboard-and-kpi-tiles

#### What Happened

Story 6.1 was implemented directly from an implementation plan rather than through the formal `/bmad-bmm-dev-story` workflow. The implementation was complete — all 8 E2E tests passing, 13 backend unit tests, 11 frontend unit tests — but `sprint-status.yaml` was never updated from `ready-for-dev`. It remained stale until Dell noticed.

#### Root Cause

The sprint-status update gate **exists** in `dev-story/checklist.md` (lines 62-63):
- `[ ] Story Status Updated: Story Status set to "review"`
- `[ ] Sprint Status Updated: Sprint status updated to "review"`

However, this gate only activates when the dev-story workflow checklist is executed. When implementation happens outside that workflow (e.g., direct plan execution, ad-hoc coding sessions), the checklist is never loaded, and the status update is skipped.

**This is a repeat of the Epic 3 retro insight: "Documented is not enforced."** The gate is documented in a checklist, but it's not structurally enforced outside the workflow that owns it.

#### Resolution Options

| # | Option | Type | Trade-off |
|---|---|---|---|
| 1 | Add explicit "Update sprint-status.yaml" instruction to the plan-mode exit protocol | Lightweight | Depends on plan authors remembering |
| 2 | Add a CLAUDE.md instruction: "After completing any story implementation, always update sprint-status.yaml" | Structural | Always loaded into context, workflow-independent |
| 3 | Create a pre-code-review gate that checks sprint-status.yaml matches the story being reviewed | Enforcement | Catches it at review time, not at dev time |

#### Recommended Action

**Option 2 + 3 combined.** Option 2 catches it at dev time (low cost, always in context). Option 3 catches any misses at review time (safety net).

#### Status: RESOLVED — Option 2 + 3 implemented on 2026-02-28

**Changes made:**
1. **CLAUDE.md** — Added "Implementation Gate — Sprint Status (MANDATORY)" section. This is always loaded into context regardless of workflow, making the gate workflow-independent.
2. **code-review/checklist.md** — Added "Sprint Status Gate" as a pre-review checkpoint (line 4). Reviewer must verify sprint-status.yaml shows `review` before proceeding. Explicitly calls out the bypass scenario.
