## Key Successes

1. **Snapshot architecture enabled clean, fast epic** — All 5 stories read directly from `calculation_snapshots`. No recalculation at query time. The investment in Story 4.5's snapshot service paid compounding returns across every dashboard.

2. **Component reuse across all dashboards** — `MarginHealthBadge`, `AtRiskKPITile`, `DataPeriodIndicator` created in early stories and reused consistently. No duplication, no recreation.

3. **RBAC scoping consistency** — All 7 endpoints enforce role-based access in the service layer (never route handlers). DM → own projects, DH → own department, Finance/Admin → all. Pattern never deviated across 5 stories.

4. **All code reviews single-round** — 5/5 stories approved in one review pass (with fixes applied inline). Contrast with Epic 4 where multi-round reviews occurred.

5. **130+ new tests with zero regressions** — E2E suite grew from 85 to 97. Backend and frontend suites maintained green throughout.

6. **Sprint-status bypass structurally fixed** — When the bypass was discovered in Story 6.1, it was fixed with a CLAUDE.md gate + code-review pre-check. "Structural gates, not team agreements" applied in real-time.

---

## Critical Challenges

1. **0/5 retro follow-through from Epic 5** — Down from 10/10 in Epic 5. Root cause: Stories 6.1 and 6.3 started before the Epic 5 retro was completed, and no 6-0 cleanup story was created afterward. The "cleanup-first" pattern only works if the retro precedes the next epic's start.

2. **Test gaps as dominant HIGH finding** — 4/5 stories had HIGH findings for missing or weak test assertions during code review. Missing click-through tests, weak rank ordering assertions, silent-pass guards, missing edge case tests. Code review is catching these reliably, but the goal should be fewer gaps to catch.

3. **Repeated gotchas not documented** — jsdom hex→rgb color conversion hit in both Stories 6.4 and 6.5. antd v6 deprecations (`Drawer.width`, `Statistic.valueStyle`) discovered in 6.4 but not documented for future reference.

4. **Pre-existing test failure carried through epic** — `employees.routes.test.ts` DELIVERY_MANAGER getting 200 instead of 403 on `GET /employees`. Found in Story 6.3, never resolved. Contradicts "inherited failures are still failures" agreement.

5. **Story 6.1 implemented outside dev-story workflow** — Led to sprint-status bypass. Fixed structurally, but highlighted that process gates only fire when the workflow runs.

## Key Takeaways

1. **Snapshot architecture enabled a clean, fast epic** — Direct row reads, no recalculation. Compounding returns on Epic 4 investment.
2. **Retro follow-through collapsed without structural enforcement** — 0/5 items completed because retro happened after epic started. Fix: enforce retro-before-epic gate.
3. **Test gaps remain the dominant code review finding** — 4/5 stories had HIGHs for missing tests. Fix: strengthen dev-story test completeness gate.
4. **Document gotchas once, not twice** — jsdom, antd v6, BigInt patterns should be shared knowledge immediately.

## Team Performance Summary

Epic 6 delivered 5 stories covering 6 dashboard views (Project, Executive, Practice, Department, Company, Employee), 7 API endpoints with role-based access control, and a model-aware Ledger Drawer — all sourced from the `calculation_snapshots` architecture established in Epic 4. The epic added 130+ tests (55 backend, 35 frontend, 41 E2E) with zero regressions and all 5 code reviews passing in a single round. The primary lesson: structural gates work when they fire, but the "cleanup-first" pattern requires the retro to happen before the next epic starts — a gap that will be closed with a new structural gate (AI-1).
