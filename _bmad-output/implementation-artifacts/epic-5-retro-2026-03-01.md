## Key Successes

1. **Cleanup-First Pattern Proved** ‚Äî Story 5-0 resolved ALL Epic 4 debt (10/10 items) before any new feature work began. This is the first epic to start from a completely green baseline. The pattern of "first story = retro cleanup" should be permanent.

2. **Architectural Clarity: Atomic vs Row-Level** ‚Äî Two distinct upload models (atomic for timesheet/billing, row-level for salary) were cleanly separated throughout the stack. Never conflated. The architecture doc's guidance on "never mix patterns" was followed precisely.

3. **SSE Integration End-to-End** ‚Äî EventEmitter ‚Üí SSE endpoint ‚Üí `useUploadProgress` hook ‚Üí cache invalidation. No external broker needed. Simple, testable, and working. The reconnect logic (one retry after 3s) handles transient failures gracefully.

4. **E2E Growth: 67 ‚Üí 85** ‚Äî 18 new E2E tests including 9 upload-center scenarios and Chain 8 (cross-role timesheet upload validation). Dynamic XLSX generation in E2E tests is a new pattern that enables runtime-dependent test data.

5. **Single-Round Code Reviews** ‚Äî All 4 stories passed code review in a single round. Contrast with Epic 4 where Story 4-5 required 2 rounds. The structural gates embedded in Story 5-0 are paying dividends.

6. **100% Retro Follow-Through** ‚Äî First epic to achieve 10/10 on previous retro action items. The "bundle as Story X-0" pattern is proven.

## Critical Challenges

1. **Story 5-3 Was Disproportionately Large** ‚Äî 10 tasks, 20+ unit tests, 9 E2E tests, 4 HIGH review findings. It transformed a single-zone page into a full 3-zone hub while adding SSE, backend-driven history, and DataPeriodIndicator. Front-end integration stories have consistently been the largest and most finding-heavy across all epics.

2. **Deferred H2: Download Template Scope** ‚Äî The "Download Template" link points to the salary template for all upload zones. Deferred with Dell's authorization, but it's a UX gap that Finance users will encounter when they look for timesheet/billing templates.

3. **Tier 3 E2E Gaps Are Structural** ‚Äî FR18.6 (timesheet rejection negative path) and FR21.3 (dashboard auto-refresh after recalc) cannot be tested at Tier 3 until their dependent surfaces exist. Unit test coverage is solid, but E2E gap tracking creates the illusion of missing coverage.

4. **Employee ID Format Concern** ‚Äî Story 5-1 noted that timesheet employee lookup uses UUIDs per spec (`WHERE id IN (...)`), but Finance users creating timesheets may prefer employee codes (EMP001). Product decision pending.
