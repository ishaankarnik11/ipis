## Successes & Strengths

### 1. Cleanup-First Pattern Proven
The Story X.0 series (3.0, 3.0b, 3.0c) was the most impactful part of the epic. Starting with retro actionables, E2E infrastructure, and bug fixes established a clean foundation for all subsequent stories. This pattern, institutionalized in Epic 2's retro, delivered measurable value.

### 2. E2E Testing Infrastructure Established
Epic 3 took the project from zero E2E tests to 47 Playwright tests covering authentication, user management, employee CRUD, bulk upload, system config, project CRUD, approval workflows, and team roster. This is a foundational investment.

### 3. TOCTOU Race Condition Fix
Story 3.1 discovered and fixed a time-of-check-time-of-use race in project status transitions. The `atomicStatusTransition()` helper became the pattern for all status-changing operations — a genuine architectural improvement.

### 4. Code Review Process Matured
9 code reviews with consistent HIGH/MEDIUM classification. The review process caught real bugs (CTC prepopulation, RBAC gaps, TOCTOU race) that would have shipped otherwise.

### 5. antd v6 Patterns Established
4 stories dealt with antd v6 compatibility (DatePicker, Select strict mode, form validation). The team developed reliable patterns for these components that will carry forward.

## Challenges & Growth Areas

### 1. CRITICAL: Silent Data Loss (Architecture Gap)
**The defining failure of Epic 3.** The `CreateEditProject.tsx` form collects model-specific fields (`slaDescription`, `vendorCostsPaise`, `manpowerAllocation`, `budgetPaise`) that are silently discarded because neither the Zod schema nor the Prisma schema have corresponding fields. The UI shows "success" while data vanishes.

- Discovered in Story 3.3 code review as HIGH finding (H2)
- NOT resolved during Epic 3 — classified as "noted" rather than fixed
- UI-only E2E tests passed because they didn't check the database
- **Root cause:** No structural enforcement of field-level data traceability

### 2. HIGH Findings Waved Through as "Noted"
Multiple HIGH severity findings across stories were closed as "Noted" without being fixed. This practice allowed the data loss bug to persist. The word "Noted" was used as a resolution status for findings that required actual code changes.

### 3. E2E Tests Didn't Verify Database State
All 47 E2E tests verified UI behavior (success notifications, page redirects, table rows) but none verified that submitted data was actually persisted to the database. This created a false sense of security where "all tests pass" masked the data loss bug.

### 4. Error Handling Gaps
5 out of 9 stories had error handling findings in code review — missing try/catch blocks, generic error messages, unhandled edge cases. This is a recurring pattern from Epic 2 that wasn't structurally addressed.

### 5. RBAC Gaps
3 stories had role-based access control gaps: missing authorization checks for certain roles, DEPT_HEAD not properly scoped to their department, Finance role access inconsistencies.

### 6. E2E Test Data Isolation
3 stories had issues with test data pollution between spec files. Fixed in Story 3.6 with test data isolation patterns, but the issue recurred multiple times.

## Team Agreements

1. **"Documented is not enforced"** — Process changes go into workflow files with structural gates, not just docs or retro notes
2. **"UI green ≠ System green"** — Every data mutation must be verified at the DB level in E2E tests
3. **"No silent data loss"** — If a field appears in the UI, it must have a Prisma column or the field is removed
4. **"HIGHs don't sleep"** — HIGH severity findings are resolved same-story, never deferred as "Noted"
